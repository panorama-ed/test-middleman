Panorama Style Guide

Introduction

As Panorama continues to grow into new product areas one challenge we are faced with is learning how to refine the Panorama brand identity and apply it cohesively to each of our products. We created this styleguide to act as a central location where we house a live inventory of UI components, brand guidelines, brand assets, code snippets, and developer guidelines.

This guide is currently tied to the NDS work, but it also contains styles that we use in our reports product.

The only way we can keep this guide relevant and useful is if all members stay familiar with this styleguide and help ensure that it is kept up-to-date.

Guiding Principles

  1. Squad members stay familiar with this guide and have the agency to keep it up to date.
  2. Engineers try to reuse elements before creating new elements.
  3. Designers try to reuse elements before creating new elements.
  4. Squad members recognize the style guide must evolve and change to meet the requirements of an evolving product.

Icons

Our icons are kept up to date in a font file that we host.

Designers primarily update the font file when we want to include new icon(s). The intructions are as follows:

  1. To update the icon font file first go to icomoon.io and click the "IcoMoon App" button in the upper right of the screen.
  2. Then click "Import Icons" in the upper left screen and upload the .svg file from our existing icon font (Note: this must be done so that no icons are lost in the update. When you go to export the new file these existing icons should still have the same characters assigned to them).
  3. Once you have uploaded the existing SVG file you should see those icons selected on the screen.
  4. Now upload your new icons, these must be in a single black color and there must be a single SVG shape for each one.
  5. When ready to export click "Generate Font", on the following screen you must assign new icons to a character that has not been used yet.
  6. Download this new font and replace all 4 existing font files.

Button Colors have Meaning

Default Button is primarily implemented in modals and the login page. Used when there is already a primary button being used for a more important action. If there are multiple high importance actions, only use secondary buttons.

<button class="btn">Default Button</button>

Primary Button are used primarily in modals and the login page. Used when we want to direct the user to take a suggested action. Reserve only one primary button in view at a time.

<button class="btn btn-primary">Primary Button</button>

White Button are primarily used in toggles and in the Summary page in the Reports platform.

<button class="btn btn-white">White Button</button>

Warning Button are used primarily in modals. The warning button has the same rules as the primary button except that it is used to warn the user they are about to take an action that cannot be reversed.

<button class="btn btn-warning">Warning Button</button>

Button Icon Indicates Action

Action Buttons are used primarily for creating groups from filters or adding a student to a list. You can add an icon to this button using a Sass mixin (button-create($icon-type: "e");) and referring to the typography.scss file for glyphs.

<button class="btn add-icon">Action Button</button>

Loaders Indicate Interaction

Use a Loader when something does not happen instantaneously. It is always safer to add this in case performance slows down as we scale.

Button Size Indicates Importance

Large buttons are used in modals and in situations when the user must use the button to do a single action on the page. Small buttons are used when there are multiple user actions on a page so they don't need to call as much attention to themselves.

<button class="btn">Large Button</button>
<button class="btn btn-small">Small Button</button>
<button class="btn btn-xsmall">Extra Small Button</button>

Toggle Buttons Change/Select Data

Toggle buttons are used primarily when toggling between marking periods and school years. 

<button class="btn-toggle">Q1</button>
<button class="btn-toggle">Q2</button>
<button class="btn-toggle active">Q3 (current)</button>

Toggle buttons can also be used to select multiple pieces of information. Use these instead of checkboxes when the information should be very prominent.

<button class="btn-toggle">2x10 relationship building</button>
  <button class="btn-toggle active">Organization session</button>
  <button class="btn-toggle active">Homework club</button>

Copy Style Guidelines

Capitalization

Headlines

Title Case, no punctuation. 

<h1>My Groups</h1>

My Groups

Subtitles

Sentence case, no punctuation. 

<h4>Viewing all 2,020 students in this group</h4>

Viewing all 2,020 students in this group

Buttons

Title Case, no punctuation.

<button class="btn">Save as a Group</button>

Hyphens and Dashes

en-dash (U+2013: "–")

Mac shortcut: option/alt + hyphen/minus sign.

Used between numbers, e.g. 1–2 incidents.


em-dash (U+2014: "—")

Mac shortcut: option/alt + shift + hyphen/minus sign.

We don’t have a use for them in our product yet.


hyphen (U+002D: "-")

This is the normal hyphen/minus sign on keyboards.

Never use a hyphen in place of an en dash or an em dash. Never use a hyphen in the phrase “At risk”. Always use a hyphen when writing “Social-emotional learning”.


Gender-neutral language

We only use gender neutral language. Do not say “his” and “her”, instead say “their”. We don’t ever want to assume a gender for a student even if we have their demographic gender attribute, as it may not be self-reported.


Number formatting

Always use comma for digit grouping, e.g. 10,000,000.


Possessive pronouns: "Your" vs "My"

Student access

For student access to the platform, we use first person for questions we will answer (e.g. "What is my daily attendance?") and second person for the answers (e.g. "Your daily attendance is not available").


Educator access

For educator access to the platform, we use the 3rd person plural pronoun, e.g. “What is their daily attendance?” (see Gender-neutral language above).
